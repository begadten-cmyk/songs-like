You are Replit Agent. Build a small production-ready web app called “SongsLike” that lets a user type a song, select it from live search results, then see similar songs. Stack and constraints below. Do everything end-to-end and stop only when it runs in the Replit preview.

BACKEND (Flask)
- Implement small auth cache for the Spotify token with Client Credentials flow.
  - Token endpoint: https://accounts.spotify.com/api/token
  - grant_type=client_credentials
  - Cache access_token and expires_at (epoch). Refresh 30 seconds before expiry.
- Helper `sp_get(path, params=None)`:
  - Prefix base URL https://api.spotify.com/v1
  - Adds Authorization: Bearer <token>
  - Timeout 10s
  - On non-200 -> raise for status with clear log; caller will catch.
- Routes:
  1) GET `/health` → {"ok": true}
  2) GET `/` → render `templates/index.html`
  3) GET `/api/search?q=<string>`:
     - Calls `/search` with type=track, market=US, limit=10.
     - Return minimal fields: id, name, artists (comma string), album name, image (medium), preview_url, external_url.
  4) GET `/api/recommendations?track_id=<id>`:
     - First try Spotify `/recommendations` with seed_tracks=<id>, limit=12, market=US.
     - If that call 4xx/5xx or returns empty, do fallback:
       a) GET `/tracks/{id}` → extract first artist id A
       b) GET `/artists/{A}/related-artists` → take up to 5 related artist IDs; if none, use A
       c) For each artist ID, GET `/artists/{id}/top-tracks?market=US`, append tracks
       d) De-duplicate and drop the seed track id, keep first 12
     - OPTIONAL features: Try to GET `/audio-features?ids=<comma ids>` in chunks of ≤100. If 403/429 occurs, swallow and continue with features omitted. Map returned features by id and attach if present; do not block rendering if missing.
     - Return shape: {"items":[{ id, name, artists, album, image, preview_url, external_url, features:{tempo,energy,valence,danceability}? }]}
- Logging:
  - Print concise DEBUG lines for failures but never crash the server.
- CORS:
  - Not required since front end served by Flask, but allow same-origin.

FRONTEND (index.html + static/app.js + static/styles.css)
- Simple, mobile-friendly page with a dark theme.
- Header: “SongsLike — Find songs that feel the same”.
- Search bar at top with placeholder “Search a song…”.
- As the user types, debounced (300ms) call to `/api/search?q=...`. Show a dropdown list with “Select” buttons.
- When “Select” is clicked:
  - Clear dropdown, show a “Recommendations” section with a skeleton/loading state.
  - Fetch `/api/recommendations?track_id=...`.
  - Render a 3-column responsive grid of cards (stacked on mobile).
- Card content:
  - Cover image (square). If missing, show a gray placeholder.
  - Title and artists.
  - Buttons/links:
    - “Open in Spotify” → external_url
    - If preview_url present, show a small <audio> tag with controls.
  - Feature chips:
    - Show chips only when feature values exist.
    - tempo: round to integer BPM
    - energy/valence/danceability: show with 2 decimals
- Empty/error states:
  - If recommendations array empty, show: “No similar songs found. Try another track.”
  - If API fails, show: “Could not fetch recommendations (code). Try again.”
- UX polish:
  - Loading spinner on search and on recommendations.
  - Keyboard support: Enter selects first search result.
  - Escape key closes the dropdown.
